## 摘要
DeQL（决策查询语言）扩展了SQL以表达决策查询：给定从关系数据中提取的选项、策略约束和可衡量的目标，DeQL查询计算出最佳行动方案。两个构造承载了扩展：CREATE CANDIDATES，它定义了从关系源中获取选项的空间；以及DECIDE，它声明了决策变量、命名约束以及针对这些变量的目标。该设计遵循SQL的原则：用户说明要优化什么，而引擎选择如何解决它；每个查询消耗并产生关系；问题的结构对引擎保持可见。本文档规定了该语言（其设计原则、语法、形式文法及执行模型），并提供了涵盖子集选择、分配、指派、调度以及多级聚合决策的示例，以及面向不确定性优化、内联模型评分和基于时间与质量限制求解的扩展。这是该规范的第一个版本；该语言正在积极开发中，本版本固定了后续修订将基于的核心构造。

## Abstract
DeQL (Decision Query Language) extends SQL to express decision queries: given options drawn from relational data, constraints from policy, and a measurable objective, a DeQL query computes the best course of action. Two constructs carry the extension: CREATE CANDIDATES, which defines the space of options from relational sources, and DECIDE, which declares decision variables, named constraints, and an objective over them. The design follows SQL's principles: the user states what to optimize while the engine chooses how to solve it, every query consumes and produces relations, and the structure of a problem stays visible to the engine. This document specifies the language (its design principles, syntax, formal grammar, and execution model) with examples spanning subset selection, allocation, assignment, scheduling, and decisions at multiple levels of aggregation, and extensions for optimization under uncertainty, inline model scoring, and time- and quality-bounded solving. It is the first version of the specification; the language is under active development, and this version fixes the core constructs on which later revisions will build.

## 1. 论文的核心问题与整体含义（研究动机和背景）
- **核心问题**：SQL 擅长描述性查询（“发生了什么”）和预测性查询（“将发生什么”），但无法直接表达**规范性（prescriptive）查询**，即基于关系数据中的选项、策略约束和可测目标来计算出最佳行动方案（如选择哪些产品、分配多少资源、调度任务时间等）。
- **研究动机**：传统做法需要用户将决策问题手工翻译为数学规划模型并调用外部求解器，过程繁琐、易错，且与数据库隔离。现有语言如 AMPL、Pyomo、JuMP 侧重于表述已提取的参数，并非直接操作关系数据。
- **整体含义**：DeQL（Decision Query Language）作为 SQL 的扩展，旨在**将决策优化与关系数据管理深度融合**，使得用户仅需声明“要优化什么”，引擎自动选择求解方法，并保持结果与 SQL 完全兼容（关系闭包）。

## 2. 论文提出的方法论：核心思想、关键技术
- **核心思想**：继承 SQL 的声明式原则、关系闭包、结构保留、保守扩展。
- **两个主要构造**：
  - **CREATE CANDIDATES**：定义决策域，包括 `DECISION KEY` 标识每个决策实体、可选的模式级 `DECISION COLUMNS` 和 `CONSTRAINTS`、`FOREIGN DECISION KEY` 实现多粒度引用，以及 `CREATE ABSTRACTION` 定义聚类抽象以支持近似求解。
  - **DECIDE 语句**：声明决策变量（`DECISION COLUMNS`）、约束（`SUBJECT TO`）、目标（`MINIMIZE`/`MAXIMIZE`）。支持：
    - **变量类型**：`BINARY`、`INTEGER`、`CONTINUOUS`，可加 `SELECTION` 修饰符（自动乘、自动链接、自动边界），还可通过 `BY` 子句控制变量粒度（如按区域、按空组）。
    - **约束类型**：聚合约束（SUM、COUNT等）、分组约束（`BY col_list`）、过滤约束（`FILTER (WHERE ...)`）、每行约束、窗口约束（`LAG`/`LEAD`/`FIRST_VALUE`/`LAST_VALUE`）、调度全局约束（`DISJUNCTIVE`、`CUMULATIVE`）。
    - **多粒度决策**：通过连接多个候选集，变量用 `ON <alias>` 指定所属表，实现不同层次变量（如批次级 start_time 与项目-批次对级 assigned）的联合优化。
    - **扩展**：随机约束（`STOCHASTIC`、`WITH PROBABILITY`、`EXPECTED`）、内联模型评分（`PREDICT`）、质量修饰符（`WITHIN` 和 `TIMEOUT`）。
- **执行模型**：三段 IR 架构（抽象 IR → 具体 IR → 物化 IR）。抽象 IR 保留符号结构；决策规划器通过结构检测选择合适的数学形式（如网络流、MILP）；物化 IR 将数据代入生成系数矩阵，交给后端求解器。输出为标准关系，可被 SQL 继续查询。

## 6. 论文的主要结论与发现
- DeQL 的核心构造（CREATE CANDIDATES + DECIDE）能够**声明式地表达多种经典决策问题**，包括：子集选择（背包）、连续分配（食谱问题）、混合整数（生产计划）、指派（工人-班次匹配）、多粒度批调度、设施选址、含 QoS 的多资源 GPU 调度等。
- 该语言**兼容 PaQL**，且扩展了分组约束、过滤约束、多粒度连接、连续变量等超越包查询的能力。
- 设计上保持了**问题结构可见**，使得引擎可以识别特殊结构（如完全幺模性）并选择高效算法，而非强制使用通用 MILP 求解器。

## 7. 优点
- **声明式与易用性**：用户只需描述问题（变量、约束、目标），引擎负责求解，类似 SQL 的“What to do”而非“How to do”。
- **关系闭包**：决策结果本身是关系，可被 SQL 继续查询、连接，或作为下一阶段决策的输入，实现组合性。
- **结构透明**：`DECISION KEY`、`BY` 子句、`FOREIGN DECISION KEY` 等显式表达问题粒度与连接，便于优化器识别子结构并选用专用算法。
- **灵活性与扩展性**：支持多粒度联合决策、随机优化、ML 模型内联评分、时间/质量受限求解等实用扩展。
- **向后兼容**：所有 PaQL 查询均可直接改写为 DeQL，保留已有生态。

## 8. 不足与局限
- **当前版本未实现的功能**：
  - **多阶段随机优化**（含 recourse 的决策序列）不支持。
  - **延迟约束生成**（如 TSP 子回路消除）无法表达，需要求解器回调。
  - **多目标优化**（如 Pareto 前沿）不支持。
- **NULL 处理严格**：数据中出现的 NULL（系数、目标项、边界）会直接报错并指出违规行，必须用 `COALESCE` 或过滤处理。
- **部分语法尚未明确定义**：例如 `FILTER` 中谓词的粒度比聚合表达式更细时的语义暂不支持，未来可能扩展。
- **依赖外部求解器**：语言本身不规定求解算法，最终性能取决于规划器选择的后端，可能存在求解不匹配或不稳定的风险。
- **无实验验证**：目前仅为语言规范，缺乏系统实现和性能评估，实用性有待后续工作证明。
